# `toolcall-repair-bench`

> ### ⚠️ Archived — read-only. This benchmark disproved its own premise.
>
> It was built to measure how well libraries *repair* malformed tool calls from small
> local models. Follow-up testing showed the "malformed tool call" problem it targets is
> **largely a capture-endpoint artifact, not a model behavior** — see
> [**The finding**](#the-finding-the-malformed-shapes-are-a-serving-stack-artifact)
> below. On every serving path a real client actually uses, current local models emit
> clean `tool_calls` and a repair layer is a no-op. It is kept public and read-only as an
> honest record of that result (and of the scoring methodology, which holds up).
> **Do not cite it as a measure of model or library quality.**

A tool-agnostic, vendor-neutral benchmark for local-model tool-call repair — and a
worked example of an eval catching that its own premise was too broad.

Small local models (Gemma, gpt-oss, and others) *can* emit a tool call in a shape a client cannot parse — but, as [The finding](#the-finding-the-malformed-shapes-are-a-serving-stack-artifact) documents, that happens on raw-completion endpoints that skip parsing, not on the chat endpoints real clients use. Several libraries try to repair such outputs. This repo measures how well they recover the intended call on a frozen corpus of such raw outputs. It is a dataset plus a scorer, not a tool, proxy, router, or agent.

## The finding: the malformed shapes are a serving-stack artifact

The corpus is built from raw outputs captured via Ollama's **`/api/generate`** — the raw text-completion endpoint, which takes **no `tools` parameter and runs no tool-call parser**. It recorded each model's native tool syntax *as plain text*, before any parser touched it. That is the origin of every "malformed" shape in the taxonomy (`json_in_content`, `xml_tool_code`, `toolName{...}`, and the rest).

On the paths a real client (OpenCode, Aider, any OpenAI SDK) actually uses — Ollama's `/v1/chat/completions` or llama.cpp with `--jinja` — the server applies the model's chat template and parses the call **before the client ever sees it**.

**Same model (`gemma4:e4b-mlx`, the model that produced the original seed captures), same prompt — the endpoint is the only variable:**

| Endpoint | Output |
| --- | --- |
| `/api/generate` (no parser) | `{"name":"get_weather","arguments":{…}}` as raw text, no `tool_calls` field — *the corpus's `json_in_content` shape* |
| `/v1/chat/completions` (parser runs) | `finish_reason: tool_calls`, parsed cleanly, empty `content` |

**Across both stacks, streaming and non-streaming, parallel calls included:**

| Stack / path | Result |
| --- | --- |
| llama.cpp `--jinja`, non-streaming | 27/27 clean (9 models) |
| llama.cpp `--jinja`, streaming + parallel | 36/36 clean (9 models) |
| Ollama `/v1/chat/completions`, non-streaming | 3/3 clean |
| Ollama `/v1/chat/completions`, streaming + parallel | 16/16 clean (incl. the `gemma4:e4b-mlx` seed model) |

Arguments were reassembled from fragmented SSE deltas, so the streaming assembly path — the one most likely to leak a partial shape — was genuinely exercised. The repair layer this benchmark evaluates is a **no-op** on every stack tested; the repair niche is real but narrow (consumers of *raw* completions: `/api/generate`, raw llama.cpp without `--jinja`, custom loops that parse model text themselves). Evidence, scripts, and reproduction steps are in [`findings/`](findings/).

## Leaderboard

The table below is auto-generated by running every available adapter over the offline corpus with `make leaderboard`. It is ranked **precision-first**, and the column order says how to read it: **unsafe false positives first, then precision, then recoverable recall, with F1 last.** A maximalist repairer (`greedy_name_match`) wins recall and F1 by emitting a call whenever a requested tool name appears, but it fabricates calls on prose, ambiguous, and adversarial cases — so it racks up unsafe false positives and sorts to the **bottom**, not the top. False positives are dangerous: a fabricated call manufactures an action the model never committed to, and on the unsafe/ambiguous family that action runs on guessed or coerced intent. F1 can hide that fabrication risk entirely, which is why it is the last tiebreak and never the sort key. Read `Unsafe FP` and `Precision` before anything else. Full methodology is in [`EVALUATION.md`](EVALUATION.md).

Lower recoverable recall is acceptable for a conservative repairer when the alternative is fabricating calls. `local_tool_proxy` is intentionally scoped to structured/recoverable repair; it is not trying to infer broad prose intent or turn every tool mention into an action. High recall from greedy name matching should be read as over-repair unless precision and unsafe-FP counts stay clean.

## Status

⚠️ **Archived.** The corpus captures the Ollama `/api/generate` (unparsed) regime specifically — see [The finding](#the-finding-the-malformed-shapes-are-a-serving-stack-artifact). Read every shape below as "raw model text before a parser ran," not "what a client receives."

This is an initial corpus, not yet a definitive benchmark. It currently ships **117 cases**: **85 real captured** outputs and **32 synthetic, hand-authored** cases that model failure shapes commonly seen from small local models, including a batch of negative controls that must not be repaired. Every case records an explicit `provenance.kind` of `captured` or `synthetic`; synthetic cases are clearly marked as illustrative, not captured measurements.

The 2026-06-14 captured model panel ran **6 local Ollama models** with **20 prompts/model** on the RTX 3070 VM and produced **120/120 successful raw captures**. A reviewed **80-case curated subset** from that panel is promoted into `corpus/cases/`; **43 local pending scaffolds** remain ignored under `corpus/pending/` and are not part of the public/scored corpus. Raw provenance is preserved under `captures/raw/2026-06-14/`, with the generated panel table in [`docs/captured-model-panel-2026-06-14.md`](docs/captured-model-panel-2026-06-14.md). Numeric case ID gaps are expected because not every capture was promoted. The online harness is stubbed.

The corpus deliberately spans three kinds of outcome, because telling them apart is the whole point:

- **Recoverable malformed intent** (gold = a tool call): a real call emitted in a broken shape a good repairer should recover.
- **No-tool natural language** (gold = `no_tool`): a plain answer or clarifying question; repairing it would fabricate a call.
- **Unsafe / ambiguous** (gold = `no_tool`): a call cannot be recovered without guessing — a missing required argument, an ambiguous tool choice, or a truncated call — so the correct, conservative behavior is to not repair.

## What It Does

- Freezes raw model outputs, each labeled with a gold tool call or `no_tool`.
- Scores any repair library through a thin adapter, reporting precision and recall.
- Runs hermetically: no network, no GPU, no clock, no randomness; CI in seconds.
- Treats `no_tool` cases as first-class and penalizes fabricated calls, including unsafe repairs of ambiguous or incomplete output.

## What It Does Not Do

- Run models or need a GPU.
- Execute the tool calls it recovers.
- Fork or reimplement any repair library; adapters import upstream.
- Privilege the maintainer's own tool; `local-tool-proxy` is one entrant among others.

## The Two Layers

The **offline corpus** is canonical and CI-gating: a frozen, hermetic, deterministic set of cases scored without network, GPU, time, or randomness. It is the citable artifact.

The **online harness** is secondary and clearly labeled live and hardware-bound: end-to-end agentic task completion with and without a repairer. It is currently stubbed and never gates a merge. See [`docs/adr/0002-two-layers.md`](docs/adr/0002-two-layers.md).

## Claims and Non-Claims

I built this after building [`local-tool-proxy`](https://github.com/cameronqj/local-tool-proxy) to avoid mistaking anecdotal successful repairs for general reliability. It is meant to keep me — and anyone building a repair layer — honest. With that in mind, the boundaries are explicit:

**What a score here means:**

- This is an initial/seed benchmark, not a definitive or industry-standard one.
- Scores are only meaningful for the included corpus, which is small and partly synthetic. They do not generalize to "model X" or "library Y" overall.
- The benchmark measures **repair precision and recall on static, captured outputs**: given a fixed string a model already emitted, does the repairer recover the right call, abstain when it should, and avoid fabricating?

**What a score here does *not* mean:**

- It does **not** measure end-to-end agent task success, multi-turn behavior, or latency.
- It does **not** run models, so it says nothing about how often a given model produces a given malformed shape in practice.
- It does **not** show that these shapes reach a real client. They are captured from `/api/generate`, which skips parsing; on chat endpoints (`/v1/chat/completions`, llama.cpp `--jinja`) the same models parse clean — see [The finding](#the-finding-the-malformed-shapes-are-a-serving-stack-artifact).
- A high F1 is **not** a safety claim. A repairer can top the table by fabricating; precision and the per-taxonomy fabrication (`FP`) counts are where unsafe repair shows up.

## Reproducing the leaderboard

The checked-in leaderboard was generated with the `local-tool-proxy` adapter available. The pure-Python rows (`naive_json`, `greedy_name_match`) are fully deterministic and reproduce exactly from any checkout. The `local_tool_proxy` row depends on the installed sibling library; the checked-in numbers were generated with **`local-tool-proxy` 0.1.0**, so install that version to reproduce that row. Clone both repos under the same parent directory so the sibling sits at `../local-tool-proxy`:

```bash
# 1. Clone both repos side by side.
git clone https://github.com/cameronqj/toolcall-repair-bench
git clone https://github.com/cameronqj/local-tool-proxy
cd toolcall-repair-bench

# 2. Install this benchmark (editable, with dev tools).
python3 -m pip install -e ".[dev]"

# 3. Install the sibling repair library so the local_tool_proxy adapter can import it.
#    The checked-in local_tool_proxy numbers were generated with local-tool-proxy 0.1.0;
#    install that version to reproduce them. (Optional — skip it and that adapter reports
#    "unavailable"; every other adapter still scores deterministically.)
python3 -m pip install -e ../local-tool-proxy

# 4. Validate, test, and regenerate.
make validate
make test
make leaderboard
```

Notes:

- The output is **deterministic** given a fixed corpus and adapter set: adapters and cases are iterated in sorted order, with no clock or randomness, so running `make leaderboard` twice produces no diff. The one external variable is the `local-tool-proxy` version (see above); pin `0.1.0` to match the checked-in `local_tool_proxy` row.
- An unavailable adapter (sibling not installed) is reported as `unavailable`, never as a silent zero or a crash.
- `make leaderboard` rewrites the table in this README between the `LEADERBOARD` markers and writes the raw numbers to [`corpus/results.json`](corpus/results.json).

## Adapters

- `naive_json` — plain `json.loads` on the raw output. The dumb floor.
- `greedy_name_match` — maximalist strawman: emits a call whenever a requested tool name appears in the output. High recall, fabricates on prose, low precision.
- `local_tool_proxy` — imports the `local-tool-proxy` library's parser. One entrant, no special treatment.
- `litellm`, `ollama_native`, `instructor` — planned stubs; raise `NotImplementedError`.

## Corpus

**117 cases**: 85 real captured outputs (a few seed fixtures plus a reviewed subset of the 2026-06-14 local Ollama panel) and 32 synthetic, hand-authored ones (each labeled with an explicit `provenance.kind`). Captured coverage is local-Ollama-only and concentrated — see [`corpus/README.md`](corpus/README.md) for the composition caveats, the case schema, the taxonomy of shapes, the three outcome classes, and the provenance fields each case records. `make validate` enforces the schema plus cross-case integrity (unique ids, gold names that were actually offered, and self-consistent provenance).
